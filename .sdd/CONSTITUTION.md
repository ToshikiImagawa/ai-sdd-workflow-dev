# プロジェクト原則

**バージョン**: 1.0.0
**最終更新日**: 2026-07-05
**ステータス**: 有効

## 目的

このドキュメントは、`ai-sdd-workflow` リポジトリ（Claude Code プラグイン・マーケットプレイス）の開発を統制する非交渉原則と基準を定義します。すべてのドキュメント、スキル、エージェント、スクリプト、設計判断はこれらの原則に従う必要があります。

## プロジェクト特性

本プロジェクトは一般的なWebアプリケーションではなく、**Claude Code プラグイン**（Markdown プロンプト、JSON マニフェスト、Bash/Python スクリプトで構成）のマーケットプレイスです。原則はこの特性に合わせて定義しています。

## 原則階層

```
1. ビジネス原則（最優先）
   ↓
2. アーキテクチャ原則
   ↓
3. 開発手法原則
   ↓
4. 技術制約
```

優先度が高い原則が低い原則に優先します。

## 1. ビジネス原則（最優先）

### B-001: Vibe Coding防止

**原則**: 曖昧な指示（「いい感じに」「適当に」「前と同じように」等）による仕様の暗黙的推測を排除し、仕様書を真実の源（Single Source of Truth）とする

**適用範囲**: すべてのスキル・エージェントの設計、およびプラグイン利用者への提供機能

**検証方法**:

- [ ] 曖昧な指示を検出する仕組み（vibe-detector）が機能を損なわずに維持されている
- [ ] 新規スキル追加時、仕様書なしでの実装を許容する設計になっていないか

**違反例**:

- 仕様書チェックを省略できる抜け道を持つスキルを追加する
- 「Vibe Coding防止」を謳いながら曖昧指示を許容するデフォルト動作にする

**準拠例**:

- clarify / vibe-detector スキルとの連携を前提に新機能を設計する

---

### B-002: 多言語対応（EN/JA）の一貫性

**原則**: `SDD_LANG` 環境変数によるEN/JA切り替えをすべてのテンプレート・出力で一貫させる

**適用範囲**: `templates/{en,ja}/` を持つ全スキル、CHANGELOG、README

**検証方法**:

- [ ] 新規テンプレート追加時、`en` と `ja` の両方を作成しているか
- [ ] 出力言語がテンプレート言語と一致しているか（ユーザーのグローバル言語設定で上書きしない）

## 2. アーキテクチャ原則

### A-001: Skills-First（Legacy Commands非推奨）

**原則**: 新規コマンドは `skills/` として実装し、`commands/`（legacy）は追加しない

**適用範囲**: 新規機能追加全般

**検証方法**:

- [ ] 新規コマンドが `skills/{name}/SKILL.md` 形式で実装されているか
- [ ] `plugin.json` の該当フィールドに登録されているか

**違反例**:

- 新規に legacy `commands/` ディレクトリへ追加する

**準拠例**:

- `plugins/sdd-workflow/skills/{name}/` にSKILL.mdとtemplates/を配置

---

### A-002: フックとスクリプトの責務分離（Shell/Python + Claude の2フェーズ実行）

**原則**: 静的なファイル操作（コピー・スキャン・環境変数エクスポート）はShell/Pythonスクリプトに委譲し、Claude自身は判断・生成・レビューに専念する

**適用範囲**: `sdd-init`, `constitution`, `check-spec` 等の複数ステップスキル

**検証方法**:

- [ ] ファイルの存在確認・コピー・削除等の決定的操作がスクリプト化されているか
- [ ] Claudeの処理はスクリプトの出力を読み取って判断する設計になっているか

**違反例**:

- ファイルコピーのような決定的操作をClaudeのツール呼び出しで逐次実行し、トークンを浪費する

**準拠例**:

- `scripts/init-structure.sh` がファイル操作を担い、Claudeは結果を検証・報告する

## 3. 開発手法原則

### D-001: Specification-Driven

**原則**: 仕様書（`*_spec.md`）・設計書（`*_design.md`）なしで実装しない

**適用範囲**: プラグイン自体の新機能・変更すべて

**検証方法**:

- [ ] `.sdd/specification/` に対応する `_spec.md` / `_design.md` が存在する
- [ ] 仕様書が実装前に更新されている

**違反例**:

- 口頭指示のみでスキル・エージェントを実装開始する

**準拠例**:

- Specify → Plan → Tasks → Implement のフローを遵守する

---

### D-002: ファイル命名規則の厳守

**原則**: `requirement/` はサフィックスなし、`specification/` は `_spec.md` / `_design.md` サフィックス必須というAI-SDD命名規則を維持する

**適用範囲**: `.sdd/` 配下のすべてのドキュメント生成処理

**検証方法**:

- [ ] スキルが生成するファイル名が命名規則表（CLAUDE.md参照）と一致しているか

---

### D-003: ドキュメント永続性ルールの遵守

**原則**: `task/` は一時ディレクトリであり、実装完了後に重要な設計判断を `*_design.md` に統合してから削除する

**適用範囲**: `task-cleanup` スキルおよび関連処理全般

**検証方法**:

- [ ] `task-cleanup` 実行前に設計判断の統合漏れがないか確認する

---

## 4. 技術制約

### T-001: JSON/Markdown構文の正当性

**原則**: `plugin.json` / `marketplace.json` は常に正当なJSONであり、Markdownドキュメントはリンク切れを起こさない

**適用範囲**: すべてのプラグイン設定ファイルとドキュメント

**検証方法**:

- [ ] `cat plugins/*/.claude-plugin/*.json | jq .` がエラーなく通る
- [ ] ドキュメント内の相対リンクが有効である

**違反例**:

- コンマ抜け等で不正なJSONをコミットする
- リンク先ファイルをリネーム・削除した際に参照を更新しない

**準拠例**:

- 変更後に `jq` によるJSON構文チェックを実行する

---

### T-002: plugin.json登録の徹底

**原則**: 新規エージェント・スキルを追加した場合、必ず `plugin.json` の該当フィールドに登録する

**適用範囲**: `agents/`, `skills/` への新規追加

**検証方法**:

- [ ] 新規ファイル追加後、`plugin.json` の diff に対応する登録が含まれているか

---

### T-003: 日本語出力の文字化け防止

**原則**: 日本語テキストに U+FFFD（�）や文字化けパターン（mojibake）を含めない。マルチバイト文字を途中で切断しない

**適用範囲**: 日本語テンプレート・生成ドキュメントすべて

**検証方法**:

- [ ] 出力前に文字化けパターンの混入がないか確認する

## 原則追加のガイドライン

新しい原則を追加する際は、以下を考慮してください：

### 良い原則の条件

| 条件        | 説明                         |
|:----------|:---------------------------|
| **検証可能**  | チェックリストで検証できること            |
| **明確**    | 曖昧さがなく、準拠/違反の判断が明確であること    |
| **正当化可能** | なぜその原則が必要かが説明できること         |
| **実現可能**  | チーム全体が実践できること              |
| **永続的**   | 一時的な方針ではなく、長期的に守るべき原則であること |

### 追加プロセス

```
1. 提案（Issue等で議論）
   ↓
2. チーム承認
   ↓
3. 原則ファイルに追加
   ↓
4. バージョンをマイナーアップ（例: 1.0.0 → 1.1.0）
   ↓
5. 影響を受けるドキュメントを更新
   ↓
6. /constitution validate で検証
```

## コンプライアンス

### 適用メカニズム

**自動化**:

- `session-start` フックによるバージョン整合性チェック
- `plugin-lint` スキルによるプロンプトファイル構文検査

**手動**:

- PRレビュー（`plugin.json` 登録漏れ、命名規則違反の確認）
- `/constitution validate` による仕様書・設計書のコンプライアンス確認

### 違反への対処

| 重大度    | 対応                 | 例                       |
|:-------|:-------------------|:-------------------------|
| **重大** | 即座にマージをブロック        | 不正なJSON、plugin.json未登録   |
| **主要** | 明示的な正当化が必要         | 仕様書なしでの実装               |
| **軽微** | 現在のPRまたはフォローアップで修正 | 軽微なドキュメント表現の修正          |

### 例外プロセス

原則の遵守が不可能な場合：

1. **文書化**: 該当 `*_design.md` に例外理由を記載
2. **正当化**: なぜ原則に従えないか説明
3. **レビュー**: PRレビューで承認を得る
4. **追跡**: `CHANGELOG.md` に記録
