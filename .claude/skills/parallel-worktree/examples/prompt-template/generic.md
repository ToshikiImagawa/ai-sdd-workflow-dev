> そのまま使わず、コピーして自プロジェクトの規約に合わせて編集し、`spawn.sh` に
> `--prompt-template <あなたのテンプレへのパス>` で渡してください。書き方は同ディレクトリの
> `README.md` を参照。プロジェクト固有の指示（規約・lint・PR 手順）は、このファイルのように
> ハードコードせず「リポジトリの CLAUDE.md や貢献ガイドを確認して従う」形にすると再利用しやすくなります。

あなたは並列 git worktree 上で起動された Claude Code セッションです。
このセッションは下記の単一 GitHub Issue を実装するために起動されました。

---

# 担当 Issue

- **Issue 番号**: #${ISSUE_NUMBER}
- **Issue 参照キー**: ${ISSUE_ID}
- **タイトル**: ${ISSUE_TITLE}
- **ブランチ**: `${BRANCH}`
- **worktree**: `${WORKTREE_PATH}`

このタスクは**既にユーザー承認済み**として扱ってよく、`/plan` モードに入る必要はありません。

# 実行ステップ

## 1. コンテキスト取得

```bash
gh issue view ${ISSUE_NUMBER}
```

issue 本文の背景・対象ファイル・受け入れ基準を熟読してください。本文が参照するドキュメントやルールがあれば、それも Read してください。

## 2. リポジトリ規約の把握

実装に入る前に、**このリポジトリの規約を確認して従ってください**。

- ルートおよび対象ディレクトリの `CLAUDE.md` / コントリビュートガイド / `README`
- コーディング規約・コミットメッセージ規約
- lint / test / build コマンド（CI 設定や package マニフェスト等から特定する）

規約はプロジェクトごとに異なります。**推測せず、リポジトリ内の記述を真実の源として確認**してください。

## 3. 実装

- シンプルさ優先 / 既存パターン踏襲 / 過剰な抽象化を避ける
- ドキュメントの記述を根拠に変更する前に、必ず Grep/Read で実態を確認する
- Issue 本文どおりに機械的に実装できる場合は即着手してよい
- 以下の場合は **AskUserQuestion で確認**する: Issue に記載のない設計判断を伴う / Issue が触れていない箇所へ影響する / 受け入れ基準を満たせない別アプローチを取りたい / 影響範囲が予想より大きい

## 4. 品質保証

PR を出す前に、**リポジトリの lint / test コマンドを実行し、通過させてください**。失敗時は修正してから次へ進みます。通らない場合は push せず、AskUserQuestion で報告してください。

## 5. コミット

**リポジトリのコミットメッセージ規約に従って**コミットしてください。

```bash
git add <変更ファイル>
git commit -m "<規約に沿ったメッセージ>"
```

## 6. push & PR 作成

```bash
git push -u origin ${BRANCH}
```

その後、**リポジトリの PR 作成手順に従って** PR を作成してください。PR 本文には必ず `Closes #${ISSUE_NUMBER}` を含めること。

## 7. 完了報告

PR 作成後、user に以下を短く報告してください。

- PR の URL
- 変更ファイル数 / 追加削除行数
- lint / test の確認結果
- 受け入れ基準のうち手動検証が必要な項目があれば明示

---

# 注意事項

- **他の並列 worktree セッションが同時稼働中**の可能性があります。push 時の rejected → fetch → rebase を順守してください。
- **lint / test が通らない場合は push しない**でください。修正不可能と判断した場合は AskUserQuestion で報告してください。

それでは、上記ステップに従って実装を進めてください。
